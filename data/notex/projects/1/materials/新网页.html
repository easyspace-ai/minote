1. 全局视角：CLI 工具 vs Agent Operating System

1.1 你在看什么

这份报告的素材来自 Claude Code 的 npm 包，从 cli.js.map 的 sourcesContent 提取出完整 TypeScript 源码，共 4756 个文件。市面上大部分开源 coding agent 只有少量文件，而 Claude Code 的 src/ 顶层有超过 50 个模块目录。

1.3 入口层暴露的设计意图

Claude Code 有多个入口：cli.tsx、init.ts、mcp.ts、sdk/，支持 CLI 终端、MCP 协议、SDK 调用、IDE 插件，体现平台化设计。

cli.tsx 有 fast-path 分发逻辑：

  • --version：零模块加载，直接打印版本号退出
  • remote-control：走 bridge 模式，不进主 REPL
  • daemon：走长驻进程模式
  • ps / logs / attach / kill：走后台会话管理
  • --worktree / --tmux：直接 exec 进 tmux，不加载完整 CLI

只有没命中任何 fast-path 时才加载 main.tsx，所有 import 都是 dynamic 按需加载，这是产品化思维的体现，优化高频调用的响应速度。

1.4 命令系统不是装饰品

src/commands/ 下有 101 个命令，涵盖 /mcp、/memory、/permissions、/hooks、/plugin、/skills、/tasks、/plan、/review、/agents、/compact、/doctor、/bridge、/teleport 等。命令系统是用户与系统运行时交互的控制面，也是生态的入口，统一加载 plugin commands、skill commands、bundled skills、动态 skills。

2. 引擎：主循环与 Prompt 编排

这是整份报告的地基，Claude Code 的一切能力最终都通过主循环跑出来，通过 prompt 告诉模型该做什么。

2.1 一个请求怎么跑起来的

  1. cli.tsx 分发到 main.tsx
  2. main.tsx 初始化状态、注册工具、构造 ToolUseContext
  3. 用户输入进入 query() 函数
  4. 进入 queryLoop() 主循环（while(true)）
  5. 每次迭代：压缩上下文 → 组装 system prompt → 调用 API → 处理工具调用 → 继续或退出

2.2 query.ts：1729 行的状态机

query.ts 是系统心脏，是 async generator，内部 while(true) 循环，通过 state 对象传递状态。每次迭代步骤：

  1. 上下文预处理：snip compact → micro compact → context collapse → auto compact，四道压缩机制
  2. Token 预算检查：如果 auto compact 关闭，检查是否接近硬限制
  3. 调用模型 API：发送消息、system prompt、工具定义
  4. 流式处理响应：边收边处理，完成的 tool_use block 立刻执行
  5. 错误恢复：prompt 太长尝试 context collapse drain、reactive compact；输出超限注入恢复消息；模型降级切 fallback model
  6. Stop hooks：模型停止输出后运行，决定是否继续
  7. Token budget：检查是否该继续
  8. 工具执行：批量执行，通过 runTools() 或 StreamingToolExecutor
  9. 附件注入：注入 memory attachments、skill discovery、queued commands
  10. 下一轮：组装新消息列表，回到循环开头

用状态机取代递归，避免长会话爆栈，9 个 continue 点对应不同继续原因。

2.3 Streaming Tool Execution：边收边跑

传统做法等模型完整输出再执行，Claude Code 使用 StreamingToolExecutor，模型输出时已完成的 tool_use block 就开始执行，减少用户等待时间。一次 turn 里 5 个工具调用，第一个工具可在模型生成第二个 tool_use 时就跑完。

2.4 Prompt 组装：一台精密的拼装机器

getSystemPrompt() 返回字符串数组，每个元素对应一个 section，分为两部分：

  • 静态部分（可缓存）：身份定位、系统运行规范、做任务的行为规范、风险动作规范、工具使用语法、语气风格、输出效率
  • 动态部分（按会话注入）：Session guidance、Memory（CLAUDE.md 内容）、环境信息、语言偏好、MCP server instructions、Function result clearing 说明、Token budget 说明

静态与动态部分用 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 标记隔开。

2.5 Prompt Cache 经济学

API 层支持 system prompt 前缀缓存，不变内容放前面，可变放后面，提高缓存命中率。动态 section 有缓存机制，只有 DANGEROUS_uncachedSystemPromptSection 创建的才每次重算（如 MCP instructions）。

2.6 行为规范：怎么让 AI 工程师不乱来

getSimpleDoingTasksSection() 明确告诉模型：

  • 不要加用户没要求的功能
  • 不要过度抽象，三行重复代码好过不成熟的抽象
  • 不要给没改的代码加注释和文档字符串
  • 不要做不必要的错误处理和兜底逻辑
  • 不要设计面向未来的抽象
  • 先读代码再改代码
  • 不要轻易建新文件
  • 不要给时间估计
  • 方法失败先诊断，不要盲目重试
  • 结果如实汇报，没跑过不要说跑过了

将行为规范制度化，不依赖模型临场发挥，解决常见的 AI 过度发挥问题。

3. 工具系统：42 个工具和一条治理流水线

3.1 Tool 接口设计

Tool.ts 定义完整接口，关键方法：

  • call(): 执行工具
  • inputSchema: Zod schema 输入校验
  • validateInput(): 细粒度输入校验
  • checkPermissions(): 工具级别权限检查
  • preparePermissionMatcher(): 为 Hook 模式匹配预处理
  • isReadOnly(): 是否只读
  • isDestructive(): 是否有破坏性
  • isConcurrencySafe(): 是否可并发执行
  • prompt(): 动态生成工具 prompt 描述
  • 还有 6+ render 方法处理不同 UI 场景

buildTool() 工厂函数提供 fail-closed 默认值：默认不安全串行执行，默认非只读走严格检查，"忘了就严格"避免未配置工具的安全问题。

3.2 42 个工具的分类

  • 文件操作：FileRead、FileEdit、FileWrite、GlobTool、GrepTool、NotebookEdit
  • 执行：BashTool、PowerShellTool
  • 调度：AgentTool、Task 系列
  • MCP 集成：MCPTool、ListMcpResources、ReadMcpResource、McpAuth
  • Web 能力：WebSearch、WebFetch
  • 用户交互：AskUserQuestion、SendMessage
  • 模式切换：EnterPlanMode、ExitPlanMode 等
  • 其他：SkillTool、SleepTool、TodoWrite、Config 等

每个工具都有独立 prompt.ts，动态生成描述，根据可用工具和权限模式调整。

3.3 工具执行 Pipeline

  1. 找工具：通过名字或别名找到 Tool 对象
  2. 解析 MCP 元数据：提取 server 信息
  3. Zod schema 校验：第一道参数校验
  4. validateInput(): 工具细粒度校验
  5. Speculative classifier：BashTool 提前分析命令风险，Hook 执行时并行运行
  6. PreToolUse hooks：运行所有注册 pre-hook
  7. 解析 Hook 权限结果，综合做出允许/拒绝决策
  8. 修正输入：根据权限决策修改输入
  9. 执行 tool.call()：真正跑工具
  10. 记录 analytics / tracing / OTel
  11. PostToolUse hooks：成功后处理
  12. 构建 tool_result block
  13. PostToolUseFailure hooks：失败后处理

Speculative Classifier 提前异步计算，减少用户等待弹窗时间。

4. 多 Agent 体系：分工和调度

4.1 为什么需要多个 Agent

Claude Code 有 6 个内建 Agent：

  • General Purpose Agent：通用任务执行
  • Explore Agent：纯只读的代码探索
  • Plan Agent：纯规划，不执行
  • Verification Agent：对抗性验证
  • Claude Code Guide Agent：使用指导
  • Statusline Setup Agent：状态栏配置

设计出发点：让一个 Agent 同时做研究、规划、实现、验证很难都做好，职责分离更专业。

4.2 Explore Agent：裁剪出来的只读专家

严格禁止创建/修改/删除文件，禁止改变系统状态，只能使用读操作工具，外部用户默认用 Haiku 模型（更快更便宜），探索不需要最强推理，速度优先。权限彻底隔离，避免探索阶段意外修改代码。
